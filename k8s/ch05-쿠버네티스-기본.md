# Chapter 5: 쿠버네티스 기본 개념

## 1. 쿠버네티스란?

### 1.1 정의

- 컨테이너 오케스트레이션 툴
- 수많은 컨테이너들을 알아서 관리해주는 "총괄 관리 시스템"

### 1.2 레스토랑 비유

| 총괄 셰프(K8s) 역할      | 설명                                              |
| ------------------------ | ------------------------------------------------- |
| 요리사 배치 (배포)       | 어떤 서버(노드)에서 어떤 컨테이너를 실행할지 결정 |
| 자동 복구 (Self-Healing) | 컨테이너 문제 시 자동 재시작                      |
| 인원 관리 (스케일링)     | 트래픽에 따라 컨테이너 수 자동 조절               |
| 주문 분배 (로드밸런싱)   | 요청을 여러 컨테이너로 분산                       |

---

## 2. 프로비저닝(Provisioning)

### 2.1 정의

- IT 인프라를 생성, 설정하고 사용할 수 있도록 준비하는 모든 과정

### 2.2 종류

| 종류                | 설명                              |
| ------------------- | --------------------------------- |
| 서버 프로비저닝     | 서버 설치, OS 설정, 네트워크 설정 |
| 사용자 프로비저닝   | 계정 생성, 권한 부여              |
| 네트워크 프로비저닝 | 라우터, 방화벽 설정, IP 할당      |
| 클라우드 프로비저닝 | AWS/GCP/Azure에서 리소스 할당     |

### 2.3 IaC (Infrastructure as Code)

- Terraform, Ansible, Puppet 등으로 자동화
- 장점: 속도, 일관성, 확장성, 문서화

---

## 3. 쿠버네티스 핵심 기능

### 3.1 선언적 구성 (Declarative Configuration)

- "원하는 상태"를 YAML로 정의 → K8s가 자동으로 구현

### 3.2 4대 핵심 기능

| 기능                  | 설명                         | 담당 컴포넌트      |
| --------------------- | ---------------------------- | ------------------ |
| **스케줄링**          | 컨테이너를 최적 노드에 배치  | Scheduler          |
| **자동 복구**         | 장애 시 자동 재생성          | Controller Manager |
| **자동 확장**         | CPU/메모리 기반 파드 수 조절 | HPA                |
| **서비스 디스커버리** | 고정 DNS + 로드밸런싱        | Service            |

### 3.3 스케줄링 고려사항

- 리소스 요구사항 (CPU, 메모리)
- 제약 조건 (nodeSelector, Affinity)
- 분산 및 안정성 (여러 노드에 분산)

## 4. Docker Swarm vs Kubernetes

### 4.1 비교

| Docker Swarm        | Kubernetes     |
| ------------------- | -------------- |
| 간단함, 쉬운 사용성 | 강력함, 확장성 |
| 가벼움, 소규모 적합 | 거대한 생태계  |
| 제한적 기능         | 풍부한 기능    |
| 학습 비용 낮음      | 학습 곡선 높음 |

### 4.2 선택 기준

**Docker Swarm 선택:**

- 소규모 팀 (1~5명)
- 빠른 프로토타입
- 간단한 컨테이너 관리

**Kubernetes 선택:**

- 대규모 마이크로서비스
- 높은 수준의 자동화 필요
- 장기적 확장성
- 클라우드 네이티브 도구 연동

## 5. 기본 용어

### 5.1 단계별 이해 흐름

```
1단계: 기초      Cluster, Node, Pod, Container
2단계: 워크로드   Deployment, Service, Namespace, Ingress
3단계: 스토리지   PV, PVC, ConfigMap, Secret
4단계: 운영      Scheduler, HPA, DaemonSet, StatefulSet, CRD
5단계: 보안      RBAC, Control Plane, CNI
```

### 5.2 기초 개념

| 용어          | 설명                                            |
| ------------- | ----------------------------------------------- |
| **Cluster**   | K8s가 관리하는 전체 시스템 (여러 노드의 집합)   |
| **Node**      | 클러스터를 구성하는 개별 서버 (Master + Worker) |
| **Pod**       | 배포되는 최소 단위 (1개 이상의 컨테이너)        |
| **Container** | 앱과 실행 환경을 묶어 실행하는 단위             |

### 5.3 워크로드

| 용어           | 설명                                     |
| -------------- | ---------------------------------------- |
| **Deployment** | 파드 관리 컨트롤러 (롤링 업데이트, 롤백) |
| **Service**    | 파드의 고정 네트워크 진입점              |
| **Namespace**  | 리소스를 논리적으로 분리                 |
| **Ingress**    | 외부 HTTP/HTTPS 요청 라우팅              |

### 5.4 스토리지/설정

| 용어          | 설명                                  |
| ------------- | ------------------------------------- |
| **PV**        | 클러스터에서 제공하는 스토리지 리소스 |
| **PVC**       | 파드가 스토리지를 요청하는 "청구서"   |
| **ConfigMap** | 환경 변수/설정 파일 관리              |
| **Secret**    | 민감 정보 저장 (비밀번호, API 키)     |

### 5.5 운영/자동화

| 용어            | 설명                                    |
| --------------- | --------------------------------------- |
| **Scheduler**   | 파드를 어떤 노드에 배치할지 결정        |
| **HPA**         | 파드 수를 CPU/메모리 기반으로 자동 조절 |
| **DaemonSet**   | 모든 노드에 동일한 파드 실행            |
| **StatefulSet** | 순서와 고유 ID가 중요한 파드 관리 (DB)  |
| **CRD**         | 사용자 정의 리소스 확장                 |

### 5.6 보안/네트워킹

| 용어              | 설명                                      |
| ----------------- | ----------------------------------------- |
| **RBAC**          | 사용자/서비스 계정별 권한 제어            |
| **Control Plane** | API 서버, 컨트롤러 매니저, 스케줄러, etcd |
| **CNI**           | 네트워크 플러그인 표준 (Flannel, Calico)  |
