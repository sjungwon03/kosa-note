# Chapter 11-1: 리소스 관리

## 1. 리소스 단위

### 1.1 메모리 단위 (IEC - 이진법)
| 단위 | 이름 | 값 |
|------|------|-----|
| Ki | Kibibyte | 2^10 (1,024) |
| Mi | Mebibyte | 2^20 (1,048,576) |
| Gi | Gibibyte | 2^30 (1,073,741,824) |
| Ti | Tebibyte | 2^40 |

### 1.2 CPU 단위
| 단위 | 설명 |
|------|------|
| m | millicore (CPU의 1/1000) |
| 1 | 1 CPU core |
| 100m | 0.1 CPU core |

### 1.3 십진법 vs 이진법
| 십진법 (SI) | 이진법 (IEC) |
|-------------|--------------|
| 1 MB = 1,000,000 바이트 | 1 Mi = 1,048,576 바이트 |
| 1 GB = 1,000,000,000 바이트 | 1 Gi = 1,073,741,824 바이트 |

> **K8s는 이진법 단위 사용** (컴퓨터 시스템에 정확)

## 2. requests vs limits

### 2.1 정의
| requests | limits |
|----------|--------|
| **최소 요청량** | **최대 제한량** |
| 스케줄링 기준 | OOM Kill 기준 |
| "필요한 자원" | "사용 가능한 최대" |

### 2.2 YAML 예시
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: resource-limit-pod-example
spec:
  containers:
  - name: my-container
    image: nginx
    resources:
      requests:
        memory: "64Mi"   # 최소 64Mi 요청
        cpu: "250m"      # 최소 0.25 CPU 요청
      limits:
        memory: "128Mi"  # 최대 128Mi 제한
        cpu: "500m"      # 최대 0.5 CPU 제한
```

## 3. QoS (Quality of Service) 클래스

### 3.1 QoS 종류
| QoS 클래스 | 조건 | 설명 |
|------------|------|------|
| **Guaranteed** | requests == limits | 최고 우선순위, OOM Kill 최후순위 |
| **Burstable** | requests < limits | 중간 우선순위 |
| **BestEffort** | requests/limits 없음 | 최저 우선순위, OOM Kill 최선순위 |

### 3.2 Guaranteed 설정 예시
```yaml
resources:
  requests:
    memory: "128Mi"
    cpu: "500m"
  limits:
    memory: "128Mi"
    cpu: "500m"
```

## 4. LimitRange

### 4.1 정의
- 네임스페이스 내 파드/컨테이너의 **리소스 범위 제한**
- 기본값 설정, 최소/최대 제한

### 4.2 예제
```yaml
apiVersion: v1
kind: LimitRange
metadata:
  name: limitrange-example
spec:
  limits:
  - type: Container
    default:        # limits 기본값
      cpu: "500m"
      memory: "256Mi"
    defaultRequest: # requests 기본값
      cpu: "250m"
      memory: "128Mi"
    min:            # 최소값
      cpu: "100m"
      memory: "64Mi"
    max:            # 최대값
      cpu: "1"
      memory: "512Mi"
```

## 5. ResourceQuota

### 5.1 정의
- 네임스페이스 전체 **리소스 총량 제한**
- 한 팀이 클러스터 자원 독점 방지

### 5.2 예제
```yaml
apiVersion: v1
kind: ResourceQuota
metadata:
  name: resource-quota
spec:
  hard:
    requests.cpu: "4"
    requests.memory: "8Gi"
    limits.cpu: "8"
    limits.memory: "16Gi"
    pods: "10"
    services: "5"
```

## 6. HPA (Horizontal Pod Autoscaler)

### 6.1 HPA 필수 조건
- 파드에 `resources.requests` 설정 필수
- Metrics Server 설치 필수

### 6.2 requests가 필요한 이유
- HPA는 `(현재 사용량) / (requests)`로 사용률 계산
- requests 없으면 사용률 계산 불가

## 7. Best Practices

| Practice | 설명 |
|----------|------|
| requests/limits 설정 | 모든 파드에 설정 |
| Guaranteed 클래스 | 중요 서비스는 requests == limits |
| LimitRange | ns 기본값/범위 설정 |
| ResourceQuota | ns 총량 제한 |
| 단위 명시 | Mi, Gi 명시 사용 |
