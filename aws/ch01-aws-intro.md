## 1.2 AWS 글로벌 인프라

### 인프라 구조
```
데이터 센터 → 가용 영역(AZ) → 리전(Region)
```

| 구성 요소 | 설명 |
|----------|------|
| **데이터 센터** | 실제 서버가 있는 물리적 장소 |
| **가용 영역 (AZ)** | 한 개 이상의 데이터 센터, 전력/네트워크 중복 구성 |
| **리전 (Region)** | 여러 AZ가 모여 하나의 지역 단위 (예: us-east-1) |

### 리전 선택 기준 4가지
| 기준 | 설명 |
|------|------|
| **규정 준수** | 법/규정에 따라 특정 국가 내 데이터 저장 |
| **지연 시간** | 사용자와 서버 거리 가까울수록 빠름 |
| **요금** | 리전마다 가격 다름 |
| **서비스 가용성** | 모든 서비스가 모든 리전에 제공되지 않음 |

### 글로벌 엣지 네트워크
| 구성 | 설명 |
|------|------|
| **엣지 로케이션** | 사용자와 가까운 곳에 데이터 캐싱 |
| **리전 엣지 캐시** | 엣지 로케이션과 리전 사이 캐시 최적화 |
| **CloudFront** | CDN 서비스, 전 세계 엣지 로케이션에 캐시 |

## 1.3 AWS와 상호 작용

### AWS 리소스 접근 방법 3가지

| 방법 | 설명 | 장점 | 단점 |
|------|------|------|------|
| **Management Console** | 웹 콘솔 | 쉬움, GUI 기반 | 반복 작업 비효율 |
| **AWS CLI** | 명령줄 인터페이스 | 스크립트 자동화 | 명령 구문 알아야 함 |
| **AWS SDK** | 프로그래밍 언어용 | 앱과 AWS 통합 | 프로그래밍 지식 필요 |

### CLI 예시
```bash
aws ec2 run-instances \
  --image-id ami-0eb63419e063fe627 \
  --instance-type t2.micro \
  --associate-public-ip-address
```

### 학습 순서
> 콘솔 → CLI → SDK

## 1.4 AWS 공동 책임 모델

### 개념
> **AWS와 고객이 보안을 함께 책임**

| 책임자 | 담당 영역 |
|--------|----------|
| **AWS** | 클라우드 자체 (데이터 센터, 하드웨어, 가상화, 네트워크) |
| **고객** | 클라우드 내부 (데이터, OS 패치, 사용자, 암호화, 접근 제어) |

### 비유
- AWS = **아파트 건설사** → 건물 기초, 외부 보안
- 고객 = **세입자** → 집 문단속, 귀중품 보관

## 1.5 루트 사용자 보호

### 루트 사용자
- AWS 계정 생성 시 사용하는 **이메일 + 비밀번호**
- **모든 권한** 가진 최고 관리자

### 위험
- 해킹 → 데이터 삭제, 비용 청구, 불법 사용

### 보안 강화
| 방법 | 설명 |
|------|------|
| **강력한 암호** | 문자, 숫자, 특수문자 포함 |
| **MFA** | 다중 인증 (휴대폰 앱, 물리 키) |

### 모범 사례
- 루트 사용자 **최소화** (결제 정보 변경 등만)
- 일상 작업 → **IAM 사용자/역할** 사용

## 1.6 IAM (Identity and Access Management)

### IAM 구성 요소

| 요소 | 설명 |
|------|------|
| **IAM 사용자** | 로그인 가능한 개별 계정 |
| **IAM 그룹** | 여러 사용자 묶는 단위, 권한 상속 |
| **IAM 정책** | JSON 형식 권한 문서 (Effect, Action, Resource, Condition) |
| **IAM 역할** | 사용자 이름/비밀번호 없는 특별 계정, 서비스가 임시 권한 위임 |

### 정책 예시
```json
{
  "Effect": "Allow",
  "Action": "ec2:StartInstances",
  "Resource": "*"
}
```

### 보안 모범 사례
1. 루트 사용자 MFA 설정
2. 관리자 권한 IAM 사용자 생성
3. 그룹 기반 권한 관리
4. 애플리케션 코드 → **IAM 역할** 사용 (사용자 X)

---

## 1.7 IAM 역할 vs 사용자

### 역할 (Role)
- "권한 묶음" → 서비스/사용자가 맡아서 사용
- EC2, Lambda 등이 임시 권한 받아 다른 서비스 호출

### 사용자 (User)
- 실제 사람이 AWS 로그인/API 접근용 계정

### 액세스 키
- CLI/SDK로 AWS 사용할 때 발급
- Access Key ID + Secret Access Key

### 실습
1. 역할 생성: `S3DynamoDBFullAccessRole`
2. 권한: `AmazonS3FullAccess`, `AmazonDynamoDBFullAccess`
3. EC2에 역할 부여 → S3/DynamoDB 접근 가능
